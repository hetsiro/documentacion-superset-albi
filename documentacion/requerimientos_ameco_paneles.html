📋 Reglas generales para queries y visualizaciones

⬜ NULLs siempre visibles
Hay registros con datos faltantes (aproximadamente ~300 filas sin valor en ciertos campos). Estos se almacenan como NULL en AmecoDashboards y deben mostrarse siempre — en gráficos y tablas — sin ocultarlos.

En Superset: activar "Show empty/null values" en cada chart. En tablas, la celda aparecerá vacía o con el literal NULL. No agregar filtros IS NOT NULL en ninguna query salvo que haya una razón explícita documentada.
📊 Uso de campos según disponibilidad de datos
Consultar la columna Usos del Diccionario de Datos antes de usar un campo en una query:

Usos = 0 → campo sin datos, no utilizar en ninguna query ni visualización.
Usos bajo (1–~50) → campo con muy pocos datos, se puede usar pero advertir en el requerimiento que los resultados pueden no ser representativos.
Usos alto → campo confiable, usar con normalidad.

Pendientes / Preguntas al cliente
✅ Jornada = 12 hs = 720 min (confirmado). Capacidad instalada = COUNT(DISTINCT resolutor+día) × 720 min — se cuentan combinaciones únicas (resolutor, día) para no inflar la capacidad cuando un resolutor tiene múltiples operaciones en el mismo día.
✅ Los valores en la BD están en minutos. El PPT muestra horas → al visualizar, aplicar SUM(col) / 60.0 para convertir a horas.
✅ Las columnas de tiempo ya están disponibles en AmecoDashboards: OperacionTiempoEstandar, OperacionTiempoDuracionReal, OperacionResolucionTiempoTraslado, OperacionResolucionTiempoEspera, OperacionResolucionTiemposAdministrativos.
⚠️ Datos corruptos en OperacionTiempoDuracionReal. Se detectaron 99 operaciones (de 1.802) con valores imposibles — el mayor registra 1.051.440 min ≈ 2 años. Esos 99 registros representan el 98% del tiempo declarado total, distorsionando completamente el gauge y las tablas.

Las métricas no aplican filtro por ahora para que el resultado en el dashboard deje en evidencia el problema al cliente. Una vez corregidos los datos en la app, los números serán precisos sin necesidad de cambiar las queries.

Query para identificar los registros afectados: SELECT OperacionId, OperacionResolutor, CAST(OperacionFechaInicio AS DATE) AS Fecha, OperacionTiempoDuracionReal, ROUND(OperacionTiempoDuracionReal/60.0,1) AS Horas FROM dbo.AmecoDashboards WHERE OperacionTiempoDuracionReal > 720 ORDER BY OperacionTiempoDuracionReal DESC

Objetivo: conocer la tasa de ingreso vs. resolución de OTs y Operaciones

Qué pide el cliente (verbatim del PPT)
2.1
Incluir selector OT ↔ Operaciones (el usuario puede cambiar entre ambas vistas).
4 tabs de apertura, cada uno con su propia tabla (Ingresadas / Resueltas / Tasa Finalización):

— Operaciones: Operaciones resueltas / Operaciones ingresadas en el rango

Tasa de Cumplimiento (SLA)

Objetivo: conocer la tasa de cumplimiento de los acuerdos de nivel de servicio de las OT

Qué pide el cliente (verbatim del PPT)
3.1
Desde la pestaña "Tasa de cumplimiento", abrir los valores por:
Tipo de tarea (Nivel 1) · Entidad (Nivel 2) · Zona
3.2
Tasa cumplimiento: OTs resueltas dentro del plazo / OTs resueltas totales en el rango
Días promedio de atraso (solo si tasa < 100%): suma de días atrasados de OTs resueltas atrasadas / total OTs resueltas atrasadas
⚡
Velocímetro (gauge): Tasa de Cumplimiento global (OTs resueltas dentro de plazo / OTs resueltas totales)
Columnas de AmecoDashboards que se necesitan
Criterio SLA — comparación fecha planificada vs real
OrdenTrabajoFechaInicio OrdenTrabajoFechaFin OrdenTrabajoFechaFinReal

✅ Criterio SLA confirmado: OrdenTrabajoFechaFinReal <= OrdenTrabajoFechaFin → cumplió plazo. OrdenTrabajoFechaFin es la fecha comprometida de cierre de la OT.

Estado (solo OTs resueltas/cerradas)
EstadoOrdenTrabajo OrdenTrabajoId
Dimensiones de apertura
TipoTarea SubTipoTarea ZonaOperacion

SubTipoTarea = Entidad (Nivel 2) del requerimiento. ZonaOperacion es campo de operación — ver nota en query.

⚠ Las OTs sin OrdenTrabajoFechaFinReal (~75 registros) son OTs aún abiertas. Se cuentan en OTs sin cerrar y se muestran siempre (regla general: NULLs visibles).
